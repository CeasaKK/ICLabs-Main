Why we use GSAP over CSS animations for production websites

June 2026 • 4 min read

For simple hover effects and basic transitions, CSS is fine. But for scroll-driven orchestration, physics, and timeline control, it fails. GSAP (GreenSock Animation Platform) is our default choice at Isocode Labs. Here is why.

First, ScrollTrigger. In CSS, triggering an animation based on the user's scroll position requires writing custom event listeners, tracking window.scrollY, computing element offsets, and managing state. This is messy, slow, and prone to layout thrashing. GSAP's ScrollTrigger solves this with highly optimized intersection calculations and native support for scrubbing, pinning, and parallax effects.

Second, timeline orchestration. If you want a headline to fade in, then a list of features to stagger-reveal, and finally a CTA button to slide up, CSS keyframes require hardcoding precise animation-delay values. If you decide to change the duration of the first animation, you have to manually recalculate every single delay. With GSAP timelines, we write: tl.from('.header', {opacity: 0}).from('.item', {opacity: 0, stagger: 0.1}, '-=0.2'). It is linear, dynamic, and easy to adjust.

Third, physics and custom easing. CSS offers standard ease-in or cubic-bezier. GSAP provides complex easing curves, bounce physics, elastic effects, and custom curves that make interfaces feel natural and premium. By directly updating CSS variables or transform matrices via inline styles, GSAP bypassed CSS's limitations and is fully optimized for browser rendering loops (requestAnimationFrame). It manages memory leaks and hardware acceleration automatically.

We also use gsap.matchMedia() extensively to handle responsiveness. Animating layout changes on desktop is different from mobile; with CSS, disabling or changing complex keyframes on smaller screens is a nightmare of media query overrides. GSAP lets us register media-specific animation sets that clean themselves up when resizing. Finally, debugging: GSAP's developer tools and timeline controls let us pause, reverse, or inspect animations frame-by-frame, whereas debugging CSS keyframes is a frustrating exercise in refreshing the page.

In our own builds, like the Isocode Labs landing page, we use GSAP to orchestrate the header blur, the tilt cards, the batch reveals, and the secondary ticker bar. The result is fluid motion that feels alive. We build websites people actually screenshot and send to their friends.

AI

Building an AI email campaign tool from scratch

June 2026 • 4 min read

We recently incubated and built an AI-powered email campaign tool. The core product requirement was simple: allow a user to generate, preview, and send highly customized email campaigns without leaving the builder interface.

Architecturally, we chose Next.js for the entire app. Next.js App Router allowed us to build robust React Server Components for fast page loading and API routes for handling communication with the OpenAI API. For the template engine, we integrated MJML, a compile-to-HTML engine that outputs highly responsive emails that look great on Gmail, Outlook, and Apple Mail alike. The OpenAI API is queried via serverless endpoints, prompting the model (GPT-4o) with context-specific data: brand tone, target audience, and campaign objective.

One of our biggest engineering challenges was prompt engineering for brand voice. If the user prompts the AI with a generic description like "write a sales pitch," the response is often filled with robotic buzzwords and exclamation marks. We solved this by creating a structured system prompt that parses past successful emails from the user's brand, extracts the writing style (sentence length, vocabulary density, call-to-action placement), and injects it into the prompt context.

Another major challenge was real-time preview rendering. Generating a full email HTML template using AI takes 5-10 seconds. Waiting in front of a blank screen is a poor user experience. We implemented a streaming UI using React Server Components and Server-Sent Events (SSE). The AI streams JSON back, which we progressively parse and render as a wireframe preview in real-time, showing the user exactly what is being generated word-by-word.

We also had to tackle API rate limiting and queue management. Since sending emails and calling OpenAI are asynchronous, long-running processes, we integrated BullMQ with a Redis backend. This queue structure ensures that even if OpenAI experiences latency spikes or if a campaign has 5,000 recipients, our servers process the requests reliably without dropping jobs or blocking the event loop.

What we learned building this in production is that AI is only as good as the system architecture surrounding it. A robust builder needs more than just a wrapper around GPT; it requires a deep understanding of UX, server queues, and template compilation.

Lessons from building a hospital management system

June 2026 • 4 min read

Building a marketing landing page requires focusing on visual aesthetic and page loading speeds. Building a hospital management system, however, requires a completely different engineering mindset: zero-tolerance for data loss, strict role-based access, and deep operational complexity.

We shipped a full operational system for a live clinic, covering patient appointments, billing, inventory, prescriptions, and doctor consultations. The core stack consisted of Next.js for the UI and Supabase for the database and authentication layers. Next.js allowed us to build complex, responsive dashboards, while Supabase provided us with real-time PostgreSQL subscriptions, robust row-level security (RLS), and secure auth.

In a hospital system, role-based access control (RBAC) is paramount. A doctor should see clinical consultation notes and medical history, but not the billing ledger. A receptionist needs access to appointments and billing, but not private patient health records. We built RBAC using PostgreSQL RLS policies. Every database query checks the user's role metadata directly at the database level, ensuring that even if a frontend developer makes a mistake in the UI code, unauthorized data can never leave the database.

Real-time updates were critical for patient queuing. Doctors need to see their waiting list update automatically when a receptionist checks a patient in. We utilized PostgreSQL's Write-Ahead Log (WAL) listener in Supabase to stream database changes to the client over WebSockets. When a receptionist updates a patient's status to "waiting", the doctor's dashboard updates instantly without requiring a page refresh or aggressive API polling.

Handling inventory management for drugs was another challenge. A clinic has hundreds of medicines with differing expiry dates. Every time a doctor writes a prescription, the system must atomically decrement the inventory stock for that specific batch. We solved this using PostgreSQL database transactions (rpc calls). If a transaction fails (e.g. if the item is out of stock), the entire prescription is rolled back, preventing orphaned records.

Building operational software taught us that reliability and security are the ultimate features. A pretty UI is useless if a billing transaction fails or a patient record goes missing. We brought this obsessive focus on database safety and clean code back to all our client builds.

Growth

How we help creators ship digital products in weeks

June 2026 • 3 min read

For creators, attention is their main currency. When launching a product, timing is everything. We have built a framework that allows creators to go from a product concept to a fully operational launch system in less than three weeks.

Our approach separates standard marketing landing pages from high-performance conversion systems. A product page shouldn't just look pretty; it must load in under a second, clearly explain the value proposition, handle sudden traffic spikes (thousands of visitors in minutes from a single social media post), and offer a friction-free checkout.

Our tech stack for creator launches relies on static generation (SSG) with Next.js or Astro, combined with a edge hosting platform like Vercel. By pre-rendering the landing pages to static HTML and serving them from CDN edge servers close to the user, we achieve sub-500ms load times globally.

For payments and checkouts, we integrate Stripe or Razorpay directly using serverless API endpoints. We avoid heavy e-commerce platforms like Shopify for single-product launches because they introduce bloated Javascript and unnecessary database queries that slow down page loads. Instead, we write custom checkout pages that collect the absolute minimum customer information, reducing cart abandonment.

We also build dynamic scarcity and proof components (such as "14 slots remaining" or "recent sales" tickers) powered by real-time database queries (using Supabase or Redis) that bypass static caching. This ensures the data is accurate without sacrificing the speed of static HTML.

By focusing on performance, custom checkout routes, and high-impact copywriting structures, we've helped creators ship digital products that convert traffic at double the rate of generic platforms.

Culture

Why we built our own internship platform

June 2026 • 3 min read

At Isocode Labs, we run internship programs for engineering students to help them build real-world products. However, managing 50+ interns across multiple active client projects and experimental ventures quickly became an administrative nightmare.

We tried off-the-shelf tools like Jira, Trello, and Notion, but they were either too complex for students or lacked features like custom review cycles and automated certificate generation. So, we did what any group of builders would do: we built our own internal platform.

Our custom dashboard allows interns to view their assigned tasks, submit code links for review, receive feedback from senior engineers, and track their progress through a structured curriculum.

Building this internal platform solved three problems at once. First, it streamlined our management overhead. Senior developers can review submissions and approve certificates in a single click. Second, it served as a playground for our interns. They didn't just use the platform; they helped build it, writing new components and shipping updates to their peers.

Third, and most importantly, building internal tools makes us better at building for our clients. It forces us to act as our own demanding client. When we build tools that our own team uses daily, we experience the friction points firsthand. We learn how UI decisions impact daily productivity, how data loading delays frustrate users, and how bugs disrupt workflows.

Building our internship platform reminded us that the best software is born from real, practical pain. We carry this developer-first, builder-first mindset into every system, database, and interface we design for our clients.
